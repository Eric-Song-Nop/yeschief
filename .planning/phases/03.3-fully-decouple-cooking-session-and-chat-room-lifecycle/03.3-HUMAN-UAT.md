## Test 1: legacy room adopt keeps the same business session

Goal: prove a real legacy room where `roomName === sessionId` is adopted instead of minting a second business session.

### Steps

1. Create a new session via API:

   ```bash
   curl -s "http://127.0.0.1:3300/sessions" \
     -H "content-type: application/json" \
     -d '{"recipeId":"recipe-garlic-rice"}'
   ```

2. Record the returned `sessionId`.
3. Ensure there is **no** row for that `sessionId` in `session_room_bindings`.
4. In the real LiveKit environment, create a legacy room whose name is exactly the same as the `sessionId`.
5. Call:

   ```bash
   curl -s -X POST "http://127.0.0.1:3300/sessions/$SESSION_ID/connect"
   ```

6. Repeat the same `POST /connect` once more.
7. Open `http://127.0.0.1:4173`, reload, and confirm the recovery list still shows only the original session.

### Expected Result

- Response `sessionId` stays equal to the original business session.
- Response `binding.roomName` equals the legacy room name.
- Repeated connect does not create a second business session.
- Recovery list still points to the original session only.

### Failure Signals

- A new `sessionId` is returned.
- `binding.roomName` changes away from the legacy room unexpectedly.
- Recovery list shows a second session card.
- Connect fails to attach to the legacy room.

## Test 2: room replacement / reconnect leaves only one tutor audio stream

Goal: prove real room replacement and reconnect do not create duplicate tutor audio.

### Steps

1. Open `http://127.0.0.1:4173`.
2. Start a new cooking session and click `Connect voice guidance`.
3. Wait until the UI shows `Voice connected` and the tutor audio is audible.
4. Record the current `sessionId` from the browser network panel or API logs.
5. Force room replacement by deleting the current room binding:

   ```bash
   curl -s -X DELETE "http://127.0.0.1:3300/sessions/$SESSION_ID/room"
   ```

6. In the browser, reconnect to the same session using `Reconnect` or reload the page, choose `Continue Cooking`, and reconnect voice guidance again.
7. Listen for tutor audio and inspect the LiveKit room/participants view if available.
8. Optionally repeat once more to cover both reconnect and reload recovery paths.

### Expected Result

- Browser and API continue to reference the same business `sessionId`.
- A new `binding.roomName` can be created, but the session does not fork.
- Only one tutor audio stream is audible after reconnect.
- No duplicate tutor audio remains in the room after replacement.

### Failure Signals

- Duplicate tutor audio or echo is heard.
- Two tutor participants or remote audio tracks remain active.
- UI jumps to a different session or shows a second recovery entry.
- Reconnect succeeds transport-wise but the session no longer matches the original `sessionId`.
