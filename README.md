# Yes Chief

Yes Chief is a voice-first cooking tutor built with `LiveKit` and `LiveKit Agents`.
The product goal is simple: keep a real-time voice coach in the cooking session while `api` remains the business source of truth and `webui` stays a lightweight companion interface.
The project also supports basic session recovery: after a refresh, users can restore an existing cooking session from the Recent Sessions list and continue from the same `sessionId`.

## Quick Start

Run from the repository root:

```bash
cp ".env.example" ".env"
```

Fill in these values in `.env`:

- `LIVEKIT_URL`
- `LIVEKIT_API_KEY`
- `LIVEKIT_API_SECRET`

Start the project:

```bash
docker compose up --build
```

Open:

- `http://localhost:8080` for the web app
- `http://localhost:3000` for the API

Notes:

- This repo expects your own `LiveKit` credentials. It does not bundle a self-hosted `LiveKit` stack.
- SQLite data is persisted by Docker volume in the default compose setup.
- The first `agent` image build can take longer because runtime voice/model files are downloaded during build.

## Tech Stack

- Monorepo: `bun workspaces`
- Web UI(two shot generated by Gemini): `React`, `Vite`, `Tailwind CSS`, `ShadcnUI`
- API: `Elysia`
- Agent: `TypeScript`, `@livekit/agents`
- Database: `SQLite`
- Shared contracts: `packages/shared`
- Tooling: `oxlint`, `oxfmt`

## Architecture

```text
apps/
  webui/
  api/
  agent/
packages/
  shared/
```

```text
User mic -> WebUI -> LiveKit Room -> Agent
Agent speech -> LiveKit Room -> WebUI speaker

WebUI -> API -> SQLite
Agent -> API -> SQLite
```

- `webui`: voice entry point and session companion UI
- `api`: business truth for sessions, steps, and timers
- `agent`: real-time voice orchestration and tutoring behavior

## Tradeoffs And Future Scaling

### Current tradeoffs

- `SQLite` keeps the MVP easy to run and demo, but it is not the right long-term choice for concurrent multi-user scale.
- Polling from `webui` is simpler than real-time state sync, this is just for MVP, I can definitely choose SSE.
- A single `api` service owning all business truth reduces drift, but it will become a bottleneck as session volume and features grow.

### Possible architecture and implementation changes for scale

- Replace `SQLite` with `Postgres` and introduce proper migrations, indexing, and connection management.
- Move from polling to push-based updates with WebSocket or SSE for session/timer state.
- Split session orchestration, timer scheduling, and summary generation into separate services or queue-backed workers.
